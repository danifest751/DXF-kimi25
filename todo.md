# Аудит проекта DXF Viewer

## Краткое резюме
Проект реализует DXF‑viewer с анализом резки и раскладкой деталей. Ядро выглядит устойчивым, но есть технологические пробелы: линтинг не подключён, UI монолитен, часть DXF сущностей обработана частично.

## Обновление
- 2026-02-18: аудит актуализирован и пересмотрены риски

## Область аудита
- Исходники в src/core и src/main.ts
- Сборка и тестирование (package.json, vite/vitest/tsconfig)
- Конфигурация (config.json)

## Архитектура и компоненты
- UI и приложение: src/main.ts
- Ядро: парсинг DXF, нормализация, рендеринг, геометрия, nesting, cutting
- Воркеры: парсинг DXF в фоне
- Хранилище: IndexedDB модуль для настроек и недавних файлов
- Тесты: vitest (геометрия, bbox, cutting)

## Сильные стороны
- Строгая типизация TypeScript и понятная модель данных DXF
- Чёткое разделение ядра по доменам (geometry, render, normalize, cutting, nesting)
- Парсинг в Web Worker снижает блокировку UI
- Наличие unit‑тестов для критичных модулей геометрии и расчётов
- Конфиг вынесен в config.json

## Риски и проблемы
- Линтинг: скрипт lint есть, но ESLint не подключён как зависимость и нет конфигурации
- UI‑слой в одном файле (src/main.ts > 1000 строк), сложная поддержка и тестирование
- В рендерере отсутствуют IMAGE/UNDERLAY (есть TODO‑заглушки)
- Bounding box для LWPOLYLINE не учитывает bulge, это влияет на точность bbox
- Модуль IndexedDB хранилища не используется в приложении
- Конфигурация приводится к типу без runtime‑валидации
- Исключение **/*.test.ts в tsconfig может скрывать ошибки тестового кода

## Рекомендации
- Разделить src/main.ts на модули: состояние, рендер, обработчики UI, nesting‑панель
- Добавить ESLint (или удалить скрипт lint, если он не нужен)
- Реализовать корректный bbox для bulge в LWPOLYLINE
- Либо реализовать IMAGE/UNDERLAY, либо явно скрывать эти типы в UI
- Подключить сохранение настроек и недавних файлов через store
- Добавить простую runtime‑валидацию config.json
- Определиться с ролью тестов в tsc: либо включить, либо не упоминать в include
