# Идеи по развитию проекта — взгляд изнутри кодовой базы

Мысли разработчика, который знает архитектуру, узкие места и технический долг проекта — а не только продуктовую картину.

---

## Диагноз с технической стороны

Продукт уже сильнее чем кажется снаружи. Core-engine с NFP-нестингом, common-line, batch optimizer, workspace library с Supabase — это не учебный проект. Но есть несколько мест где **архитектура сдерживает рост продукта**, и именно с них стоит начинать.

---

## Приоритет 1 — Устранить архитектурные тормоза

### 1.1 Офлайн-режим и Service Worker

Сейчас при нестинге без интернета — fallback на локальный движок, но без common-line. Это неочевидно для пользователя и порождает баги (как недавний случай с тихим fallback без тоста).

**Что сделать:**
- Progressive Web App (PWA) с Service Worker
- Кэширование core-engine и библиотеки файлов
- Явный UI-статус: "онлайн / офлайн / деградированный режим"
- Полный нестинг offline через SharedArrayBuffer + Worker

**Почему сейчас:** Архитектура уже почти готова — core-engine изолирован, api.ts абстрагирует запросы. Нужно только добавить слой.

---

### 1.2 Web Worker для нестинга в браузере

Сейчас локальный нестинг (fallback) блокирует главный поток UI. При 100+ деталях — UI зависает.

**Что сделать:**
- Перенести `nestItems` в dedicated Web Worker
- Передавать прогресс через postMessage (фазы: `preparing → computing → done`)
- UI остаётся отзывчивым во время расчёта

**Почему важно:** UI уже показывает `nestingPhase`, но сам расчёт блокирует рендер. Это не баг — это архитектурный долг.

---

### 1.3 Rate limit — умная очередь вместо отказа

Сейчас при 429 — тост и всё. Пользователь должен повторять вручную.

**Что сделать:**
- Клиентская очередь запросов с retry after X секунд
- Countdown в UI: "Следующий запрос через 42 сек"
- Автоматический повтор без участия пользователя

**Почему важно:** Лимит 10/мин — это много для одного человека, но мало для активной сессии. Очередь убирает проблему полностью.

---

## Приоритет 2 — Улучшения нестинга

### 2.1 True Shape нестинг через NFP в облаке

В коде уже есть `strategy: 'true_shape'`, но он не используется в UI. Движок поддерживает NFP-нестинг для реальных полигонов — не bbox.

**Что сделать:**
- Включить true_shape в UI как отдельный режим
- На сервере — более тяжёлый расчёт, timeout 60s вместо 30s
- Показывать разницу в utilization: bbox vs true_shape

**Почему это мощно:** Для сложных деталей (Г-образные, Т-образные) true_shape даёт +15-30% плотности. Это реальные деньги на металле.

---

### 2.2 Мульти-лист нестинг с разными материалами

Сейчас нестинг — всегда один тип листа. В реальности набор деталей разбивается по материалам и толщинам.

**Что сделать:**
- Группировка сета по `materialId` перед нестингом
- Каждая группа нестится на своём листе своего формата
- Результат — несколько пакетов листов с разными материалами

**Почему важно:** Материалы уже есть в state. Это логичное продолжение уже существующей фичи.

---

### 2.3 Нестинг с учётом направления проката

Для анизотропных материалов (сталь, алюминий) направление детали на листе критично.

**Что сделать:**
- Флаг `grain direction` на детали
- Ограничение поворота: только 0°/180° для деталей с grain
- Визуальное отображение направления на листе

---

## Приоритет 3 — Продуктовые фичи с готовой базой

### 3.1 Шаблоны наборов (Templates)

Сейчас set очищается или накапливается вручную. Для повторных заказов — боль.

**Что сделать:**
- Сохранение текущего набора как именованного шаблона
- Загрузка шаблона одним кликом
- Привязка шаблона к workspaceId в Supabase

**Почему легко:** Persist-слой уже есть (`persistState`), Supabase уже используется. Добавить одну таблицу `set_templates`.

---

### 3.2 История раскладок

Сейчас результат нестинга хранится только в `lastEngineResult` в памяти. При перезагрузке — потеря.

**Что сделать:**
- Автосохранение каждого результата нестинга по хешу
- История последних N раскладок с датой, параметрами, thumbnail
- Восстановление любой раскладки одним кликом

**Почему важно:** Share-flow через хеш уже работает. История — это просто список хешей с метаданными.

---

### 3.3 Умный split с автодобавлением в библиотеку

Split уже реализован, но требует ручных действий.

**Что сделать:**
- При split — автоматически предлагать добавить все контуры в библиотеку как новые файлы
- Группировка в каталог с именем исходного файла
- Дедупликация: не добавлять контуры которые уже есть

---

### 3.4 Экспорт в SVG и PDF

Сейчас экспорт только в DXF. Но для коммуникации с клиентом, согласования деталей — нужен PDF или SVG.

**Что сделать:**
- SVG-экспорт листа (уже есть рендер на canvas — можно сериализовать)
- PDF через `window.print()` или pdfmake с embedded SVG
- Для каждого листа: превью + таблица деталей

**Почему легко:** Рендер на canvas уже есть. SVG — это другой output того же pipeline.

---

### 3.5 Telegram: уведомления о событиях

Сейчас бот умеет только логин. Это waste — у пользователя уже есть бот в контакт-листе.

**Что сделать:**
- Уведомление когда нестинг завершён на сервере (для тяжёлых задач)
- Уведомление когда коллега загрузил файл в общий workspace
- Кнопка "Получить DXF листа" прямо в боте по хешу

**Почему важно:** Инфраструктура бота уже есть. Это несколько хендлеров и одна таблица подписок.

---

## Приоритет 4 — Технический долг который тормозит

### 4.1 Полноценные E2E тесты через Playwright

Сейчас тесты — unit и интеграционные. Нет ни одного E2E теста реального пользовательского сценария.

**Что написать в первую очередь:**
- Загрузка DXF → появление в библиотеке
- Добавление в сет → запуск нестинга → результат
- Экспорт листа → скачивание файла
- Авторизация через код → синхронизация библиотеки

**Почему сейчас:** Playwright уже установлен. Один упавший тест заменяет часовую ручную проверку.

---

### 4.2 Метрики и мониторинг производительности

Sentry есть, но только для ошибок. Нет данных о том:
- сколько в среднем занимает нестинг
- какой % запросов успешен
- на каком этапе падает pipeline

**Что добавить:**
- Sentry Performance трейсы для нестинга
- Custom метрики: utilization%, время расчёта, размер набора
- Алерт если p95 нестинга > 20s

---

### 4.3 CDN для статики и WebAssembly

Если core-engine компилируется в WASM — это большой бинарник который грузится при каждом открытии.

**Что сделать:**
- Cache-Control: immutable для хешированных бандлов
- Vercel Edge Cache для статики
- Preload hint для WASM модуля

---

## Что я бы не трогал сейчас

- **Полный CAD-редактор** — слишком дорого, и есть LibreCAD/Inkscape. Лёгкая правка (move/delete/snap) — да, но не constraints и не dimensions.
- **G-code и CAM** — другая область, другая аудитория, другие стандарты.
- **Мобильное приложение** — PWA покроет 80% кейсов на телефоне за 10% усилий нативного приложения.
- **Свой платёжный шлюз** — пока нет регулярных платящих пользователей, это overhead.

---

## Самая сильная техническая связка

Если думать не о фичах, а о технических инвестициях с максимальной отдачей:

**Web Worker для нестинга + PWA offline + история раскладок + шаблоны наборов**

Это четыре изменения которые:
1. Не меняют бизнес-логику
2. Резко улучшают UX
3. Делают продукт надёжнее
4. Основаны на уже существующей архитектуре

И именно они уберут основные точки трения которые я вижу в коде прямо сейчас.
